# 🗺️ DineEvent 멀티 테넌시 SaaS 서비스 전환 마스터 로드맵

싱글 매장용 이벤트 룰렛 시스템(`new_event-roulette`)을 여러 매장이 동시에 개별 가입 및 운영할 수 있는 **멀티 테넌트 SaaS형 서비스**로 안전하게 진화시키기 위한 아키텍처 청사진과 실행 로드맵입니다!

---

## 📋 멀티 테넌트 서비스 전환을 위해 해야 할 내용 전부

멀티 테넌시 구현을 위해 수정 및 개발해야 할 핵심 영역은 크게 **5가지**입니다.

### 1. URL 및 테넌트 식별 (Routing)
* **목표**: 접속한 URL을 통해 "어느 매장의 화면인가?"를 클라이언트(React)가 즉시 파악하는 것입니다.
* **구현 방식 (URL Path 방식 추천)**: 
  * 사용자 화면: `eventroulette.com/:tenantId` (예: `/starbucks`, `/burgerking`)
  * 관리자 화면: `eventroulette.com/:tenantId/admin`
* **작업**: `App.jsx` 내의 React Router 구조를 `:tenantId` 동적 파라미터가 최상위에 들어오도록 전면 개편합니다.

### 2. 테넌트 컨텍스트 (React Context)
* **목표**: 현재 사용자가 접속 중인 매장 ID(`tenantId`)와 매장 기본 정보를 앱 내부의 모든 컴포넌트가 편리하게 공유하도록 만드는 것입니다.
* **작업**: `TenantContext`를 설계하여 라우터의 `:tenantId` 값을 받아오고, 해당 매장의 영업 정보(매장이름, 테마 컬러 등)를 최초 로드하여 전역 상태로 유지합니다.

### 3. 데이터베이스 (Firestore) 아키텍처 재설계 (가장 중요)
* **목표**: 각 매장의 데이터(공지사항, 룰렛 세팅, 당첨자 목록 등)가 섞이지 않도록 격리하는 것입니다.
* **추천 방식 (하위 컬렉션 아키텍처 - Subcollection)**:
  * 기존 단독 컬렉션이었던 구조를 특정 테넌트 하위로 이식합니다.
  * 기존: `/notices/{noticeId}`
  * 변경: `/tenants/{tenantId}/notices/{noticeId}`
  * *이유*: 보안 규칙(Security Rules)을 쓰기 매우 편리하며 매장 간 데이터가 물리적으로 명확히 분리됩니다.

### 4. 어드민 인증 및 보안 규칙 설계 (Authentication & Security)
* **목표**: A 매장 사장님이 B 매장의 어드민 페이지에 주소를 치고 강제 접근하거나 데이터를 조작하는 것을 원천 차단하는 것입니다.
* **작업**: 
  * Firestore 내에 `/tenants/{tenantId}/admins/{userId}` 구조를 만들어 특정 사장님이 관리할 수 있는 매장 권한을 매핑합니다.
  * Firebase Security Rules에 `request.auth.uid`가 해당 테넌트의 `admins` 컬렉션에 존재하는지 엄격히 확인하는 구문을 빌드합니다.

### 5. 테넌트 생성 시스템 (Provisioning)
* **목표**: 새로운 매장 가입 시 자동으로 해당 매장에 필요한 기본 룰렛 아이템, 기본 문자 템플릿 등의 초기 데이터를 알아서 생성해 주는 자동 세팅 기능입니다.

## 🏁 어떤 순서대로 개발해야 할까요? (추천 실행 로드맵)

모든 것을 한 번에 고치면 에러를 잡기 힘듭니다. **화면(라우터) → 뼈대(컨텍스트) → 데이터(DB) → 보안(Security) 순서**로 점진적으로 확장하는 것이 가장 과학적입니다.

### 1단계: 테넌트 식별 URL 라우팅 구현 (최우선)
* **해야 할 일**: `App.jsx`에서 `/:tenantId` 기반으로 라우팅 구조를 감싸서 URL 경로를 처리합니다.
* **목적**: `http://localhost:5173/starbucks/admin`에 들어왔을 때 `:tenantId`로 `starbucks`라는 문자열을 올바르게 뽑아내는지 확인합니다.

### 2단계: React `TenantProvider` 구축
* **해야 할 일**: `TenantContext`를 만들고, URL에서 추출한 `tenantId`를 이용해 Firestore의 해당 매장 기본 설정 정보를 읽어와 앱 전역에 뿌려줍니다.
* **목적**: 화면 상단에 각 매장의 실제 매장 이름이 올바르게 뜨는지 확인하는 단계입니다.

### 3단계: Firestore CRUD 패스 리팩토링 (본격적인 코드 수정)
* **해야 할 일**: 현재 `db`와 연동 중인 모든 API 함수(공지사항 불러오기, 룰렛 돌리기, 문자 발송하기 등)의 경로를 `tenants/${tenantId}/...`로 동적으로 변경합니다.
* **목적**: 매장별로 독립적인 공지사항과 룰렛 정보가 조회 및 수정되는지 확인합니다.

### 4단계: Firebase 보안 규칙 및 어드민 권한 제어 코딩
* **해야 할 일**: Firebase Rules를 배포하여 비정상적인 경로의 악의적 데이터 요청을 데이터베이스 자체에서 원천 거부하도록 방어벽을 칩니다.
* **목적**: 로그인한 사장님이 허용되지 않은 다른 매장의 세팅을 건드릴 때 붉은색 권한 에러(Permission Denied)가 바르게 발생하는지 체크합니다.

### 5단계: 회원 가입 및 매장 자동 생성 폼(Provisioning) 구현
* **해야 할 일**: 신규 매장 신청 버튼을 누르고 매장 정보를 적으면, Firestore에 신규 테넌트 레코드가 박히고 어드민 권한이 부여되는 자동화 워크플로우를 완성합니다.
